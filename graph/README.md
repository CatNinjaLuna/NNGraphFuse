# graph/

This directory contains the ONNX parser (`ir.py`) and the graph diff visualizer (`visualize.py`).

---

## Visualization Design: Cluster-Based Summary Graph

### Why not a standard node-link diagram?

The most literal way to visualize a computation graph is to render every node as a point and every data-flow edge as a line. For a small graph this works. For ResNet-50 at 124 nodes, or MobileNetV2 at 172, it produces an unreadable hairball: overlapping edges, illegible labels, and no clear signal about what the optimization pass actually changed.

The standard response is to apply a force-directed layout (e.g. networkx's spring layout) and hope the graph self-organizes. It rarely does for deep learning graphs, which have long chains of identical op types connected by hundreds of tensor edges.

`visualize.py` takes a different approach: the **cluster-based summary graph**.

---

### What a cluster-based summary graph is

Instead of rendering individual nodes, the summary graph **collapses all nodes of the same op type into a single representative node**. Each representative node carries two pieces of information:

- **Label**: the op type name (Conv, Relu, Add, etc.)
- **Size**: proportional to how many individual nodes that op type represents

Edges between representative nodes represent data-flow connections between op types, with edge weight proportional to how many such connections exist in the underlying graph.

The result is a compact graph with at most as many nodes as there are distinct op types — typically 8 to 10 for ResNet-50 and MobileNetV2, regardless of total graph size.

```
Full graph (ResNet-50, 124 nodes):          Summary graph (9 nodes):

Conv_0 → Relu_0                              Conv (53) → Relu (49)
Conv_1 → Relu_1                                   ↓           ↓
Conv_2 → Relu_2      collapses to →          Add (16) ←──────┘
...                                               ↓
Conv_52 → Relu_48                            Gemm (1)
```

This is not a lossy approximation — it is a structurally faithful summary. The topology of the summary graph mirrors the real data-flow topology of the model architecture (Conv feeds into Relu, Relu feeds into Add, Add feeds into the next Conv, and so on). What it discards is the repetition, which is exactly what makes full node-link diagrams unreadable at this scale.

---

### Why this works for visualizing optimization passes

An optimization pass changes the graph in one of three ways:

1. **Removes nodes** (fusion removes Relu; constant folding removes Constant)
2. **Rewires edges** (fusion rewires Conv output to bypass Relu)
3. **Leaves nodes unchanged** (dead node elimination on a clean graph)

All three changes are directly visible in the summary graph:

- A removed op type disappears entirely from the after-side diagram
- A reduced op type shrinks (smaller node, lower count label)
- An unchanged graph looks identical on both sides

The before/after side-by-side layout makes the delta immediately apparent. On the before side, nodes affected by the pass are highlighted in red (removed) or gold (modified). On the after side, the graph reflects the post-pass state in standard op-type colors.

---

### Precedent

Cluster-based summary graphs are standard in production compiler visualization tools. TensorBoard's graph view collapses repeated subgraphs into named scopes. Netron (the ONNX model viewer) offers a similar folding mode for large graphs. LLVM's opt tool prints per-pass summaries in exactly this format: op type, count before, count after.

`visualize.py` applies the same principle: show the structure that changed, not the structure that repeated.
